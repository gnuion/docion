Kubernetes

December 5, 2022 · 10 min read
Jon Zuka
Blog owner

This guide assumes a basic understanding of containers. In this blog we explore some basic concepts of kubernetes. It is helpful to have template files.

Nodes​

Nodes host applications in form of containers. Nodes can be worker nodes or master nodes.

Master Node​

The control plane in a master node hosts the kube-apiserver which contains:

  • ETCD - a database that stores information in key-value format
  • kube-scheduler - deploys containers to worker nodes
  • controller-manager - monitors and ensures the state of the cluster

Worker Node​

In the worker node the kubelet agent listens to the kube-apiserver and makes sure that the state of the node is as it should by deploying and destroying containers. The kube-proxy service runs on worker nodes and ensures container communication.

Core components​

ETCD​

When executing a kubectl command the kubeapi-server authenticates and validates requests. It then transacts with the ETCD cluster. You don't have to use kubectl. You can also directly send POST requests to the API. The scheduler monitors the information in ETCD and through the kube-apiserver it instructs the kubelet in the worker nodes to create or destroy pods. The kubelet daemon reports to the kube-apiserver which transacts the information with ETCD.

kube-spiserver​

The kube-apiserver is deployed as a pod within the kube-system namespace in the master node . To view the pods within the kube-system namespace, run:

kubectl get pods -n kube-system

You can see the options within the pod definition file locatet at:

cat /etc/kubernetes/manifests/kube-apiserver

You can view the systemd kube-apiserve service definition by running:

cat /etc/systemd/system/kube-apiserver.service

You can also see the running procces:

ps -aus | grep kube-apiserver

kube-controller-manager​

Ensures the correct state of the kubernetes cluster. It is a process that continiously monitors the state of various components and takes neccesary actions to ensure the correct state. For example, the node-controller checks the status of the nodes through the kube-apiserver every 5 seconds. If it doesn't recieve heartbeats from the node, it marks it unreachable. THe node has then 5 minutes to come back up. If it doesn't, it removes the pods from that node and rechedules them on healthy nodes.

kube-controller-manager is also deployed as a pod within the kube-system namespace:

kubectl get pods -n kube-system

You can view the kube-controller-manager options in the manifest folder:

cat /etc/kubernetes/manifests/kube-controller-manager.yaml

kube-scheduler​

The kube-scheduler only decides which pods go to which node. It doesn't actually place the pod on the node. That is the job of the kubelet which runs on each node. Depending on pods' resource requirements the scheduler finds the best nodes for them. It uses a score system to achieve that. Some of the criteria are:

  • Resource requirements and limits
  • taints and tolerations
  • node selectors/affinity

kubelet​

The kubelet manages the whole worker node.

kube-proxy​

kube-procy is a process that runs on each node in the kubernetes cluster. It looks for services and creates the rules for forwarding traefic that targets a service to the corresponding pods. It achieves this by using ip-tables rules. It translates the IP of the service to the IP of the matching pod. A service is a virtual component, it doesn't run any application by itself.

Objects​

Pod​

A pod is a single instance of the application. Pods have containers. For scaling you replicate the pods, not the containers within a pod!

Deploying a pod​

To deploy a pod, you can for example run:

kubectl run nginx --image nginx

To see the pod, you can run kubectl get pods.

Pods with yaml​

Kubernetes can use yaml files for creation of objects such as pods, replicas, deployments, services, etc. They all follow a similar structure:

# pod.yaml
apiVersionL: v1
kind: Pod
metadata:
  name: podName
  labels:
    podLabel: podLabelValue
    anotherPodLabel: anotherPodLabelValue
spec:
  containers:
    - name: containerName
      image: containerImage
apiVersion​

The apiVersion is the version of the kubernetes api you are using to create the object. A few possible values are:

KindVersion
Podv1
Servicev1
ReplicaSetapps/v1
Deploymentapps/v1
kind​

The kind refers to the type of object we want to create. Some possible values are: Pod, Service, ReplicaSet and Deployment.

metadata​

The metadata is data about the object like its name and labels.

spec​

The spec provides additional information about the object. It is specific to each kind.

Once the definition file is created run the following command to create the pod (assuming you named the definition file pod.yaml):

kubectl create -f pod.yaml

For detailed information about the pod, run:

kubectl describe pod podName

ReplicaSet​

The ReplicaSet helps us run more than one instance of the same Pod, thus providing high availability. By using a ReplicaSet we can balance the load across multiple pods which can run across worker nodes.

ReplicaSet with yaml​

# ReplicaSet.yaml
apiVersion: /apps/v1
kind: ReplicaSet
metadata:
  name: replicaSetName
  labels:
    replicaSetLabel: replicaSetLabelValue
spec:
  template:
    metadata:
      name: podName
      labels:
        podLabel: podLabelValue
        anotherPodLabel: anotherPodLabelValue
    spec:
      containers:
        - name: containerName
          image: containerImage
replicas: 2
selector:
  matchLabels:
    podLabel: podLabelValue
template​

The template is the Pod definition without the fields apiVersion and kind.

replicas​

The number of pods that should be running.

selector​

The selector identifies the Pods the ReplicaSet should manage. It does so, by matching the labels.

When ready, run the following command to create the ReplicaSet:

kubectl create -f ReplicaSet.yaml

Similar to the pods, you can run kubectl comands to inspect other objects such as the ReplicaSet.

kubectl get replicaset

Scale ReplicaSet​

To scale the ReplicaSet, that is to change the target number of running pods, first change the number of replicas in the ReplicaSet definition file. Then, run the following command:

kubectl replace -f ReplicaSet.yaml

Another way to do it is to use the same definition file, but run kubectl scale instead.

kubectl scale --replicas=6 -f ReplicaSet.yaml

or

kubectl scale --replicas=6 replicaset replicaSetName

The second method will not modify the file.

tip

Don't forget to run kubectl --help for helpful information!

Deployments​

A Deployments offers extra capabilities to a ReplicaSet that allows for seamless version swtiching.

Deployment definition file.​

The Deployment definition file is similar to the Replicaset definition file.

# ReplicaSet.yaml
apiVersion: /apps/v1
kind: Deployment
metadata:
  name: replicaSetName
  labels:
    replicaSetLabel: replicaSetLabelValue
spec:
  template:
    metadata:
      name: podName
      labels:
        podLabel: podLabelValue
        anotherPodLabel: anotherPodLabelValue
    spec:
      containers:
        - name: containerName
          image: containerImage
replicas: 2
selector:
  matchLabels:
    podLabel: podLabelValue

Similar to other objects, kubectl commands are also available for deployments.

Namespaces​

Namescpaces provide isolation. Cubernetes creates a set of pods and services for its internal purposes under the namespace called kube-system. Each namespace can have its own set of policies.Resouce limits can also be assigned to namespaces.

Connect to services across namespaces​

Within a namespace, you can reach a service simply by its name. To target a service from another namespace dev append .dev.svc.cluster.local to the service name.

kubectl with namespaces​

To target a specific namespace, append it to the kubectl commmand after --namespace= argument. Note, this also works to determine in which namespace an object should be created.

yaml definition namespace​

If you specify a namespace: under the metadata section in the definition file, by default the object will be created under that namespace.

How to create a new namespace?​

Like for any other object, you can use a namespace definition file,

## Namespace.yaml
apiVersionL v1
kind: Namespace
metadata:
  name: namespaceName

or by running:

kubectl create namespace namespaceName

Switch namespace​

To set a namespace as default for kubectl commands use:

kubectl config set-context $(kubectl config current-context) --namespace=namespaceName

Resource Quota​

To define resource limitations within a namespace, use a Resource Quota. To create a Resource Quota, you can use a definition file.

# ResourceQuota.yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: resourceQuotaName
  namespace: namespaceName
spec:
  hard:
    pods: "numberOfPods"
    requests.cpu: "numberOfCpus"
    requests.memory: "valueOfMemory"
    limits.cpu: "numberOfCpus"
    limit.memory: "valueOfMaxMemory"

Services​

Services enable the connection between groups of pods.

NodePort​

A NodePort Service maps a port on the node to a port on the Pod. The TargetPort is the port on the Pod. The port on the Service itself is refered to as the Port. The Service has its own IP address within the cluster called Service ClusterIP. The Port on the Node itself is called NodePort. The default valid range for NodePorts is 30000-32767

NodePort Service yaml file​

THe definition file for the NodePort Service looks like this:

# NodePortService.yaml
apiVersion: v1
kind: Service
metadata:
  name: serviceName
spec:
  type: NodePort
  ports:
    - targetPort: 80
      port: 80
      nodePort: 30003
  selector:
    podLabel: podLabelValue

Within the ports array, only the port is mandatory. targetPort will be assumed to be same as Port and nodePort will be chosen at random within the valid range. To link the service to a Pod, use labels with selectors.

ClusterIP​

A ClusterIP Service is a service that is only accesible within a cluster using its ClusterIP or the service name.

ClusterIP yaml file​

# NodePortService.yaml
apiVersion: v1
kind: Service
metadata:
  name: serviceName
spec:
  type: ClusterIP
  ports:
    - targetPort: 80
      port: 80
  selector:
    podLabel: podLabelValue

Scheduling​

Every Pod has a filed called nodeName that is not set by default. The scheduler schedules a pod on a node by setting the nodeName property of the Pod to the name of the node. For scheduling a pod manualy, you can set the nodeName yourself.

Binding object.​

To reschedule a pod to another node, write a binding definition.

apiVersion: v1
kind: Binding
metadata:
  name: nginx
target:
  apiVersion: 1
  kind: Node
  name: nodeName

Then, send a post requist to the binding API.

curl --header "Context-Type:application/json" --request POST --data '{"apiVersion":"v1", "kind":"Binding"...' http://$SERVER/api/v1/namespaces/default/pods/$PODNAME/binding

Networking​

Traefik Ingress​

A trafik ingress definition file looks like this:​

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: IngressName
  namespace: default

spec:
  rules:
    - host: example.com
      http:
        paths:
          - path: /
            pathType: Exact
            backend:
              service:
                name:  serviceName
                port:
                  number: 80

Storage​

Local Storage Provider.​

A Local Storage Provider enabled the ability to create persistent volume claims using local storage on the node.

Persistent Volume Claim​

apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: persistentVolumeClaimName
  namespace: default
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: local-path
  resources:
    requests:
      storage: 2Gi

Pod with local storage​

apiVersion: v1
kind: Pod
metadata:
  name: podName
  namespace: default
spec:
  containers:
    - image:imageName
      name: containerName
      imagePullPolicy: IfNotPresent
      volumeMounts:
        - name: volumeName 
          mountPath: pathOnTheContainer
      ports:
        - containerPort: portNumber
  volumes:
    - name: volumeName
      persistentVolumeClaim:
        claimName: persistentVolumeClaimName

Then, you can apply the yaml using kubectl.
